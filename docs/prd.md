# Outside — AI-Powered English Learning

## 产品定位

一个以大模型为内容引擎的英语学习平台。用户通过阅读 AI 生成的文章来学习英语，在真实语境中积累词汇，形成「阅读 → 标记生词 → 复习 → 再阅读」的自然闭环。

核心差异点：不是背单词工具，不是题库，而是一个**有语境的阅读环境**——每篇文章都是为用户当前水平和兴趣量身生成的。系统会把用户标记过的生词悄悄织入后续文章中，用新场景完成复习，像小说里的伏笔一样自然。

## 核心概念

| 概念 | 说明 |
|------|------|
| Article | AI 生成的英语学习内容，形式随主题变化（文章 / 对话 / 场景等） |
| Word | 用户在阅读中标记的生词，绑定到具体文章和句子 |
| Vocabulary | 用户的生词本，所有标记过的单词汇总，支持回溯原文语境 |
| Topic | 学习主题，分内置主题和用户自定义主题，各有独立的生成策略、内容形式和难度曲线 |
| Custom Topic | 用户自建的主题，可定义内容形式、prompt 策略、难度参数等 |
| Review Weaving | 嵌入式复习机制——系统在生成新文章时自动编织旧生词 |

## 主题体系（Topic System）

不同主题不只是"换个话题"，而是完全不同的内容生成策略、排版形式和交互方式。

### 内置主题

#### 日常口语（Daily Conversation）

- **内容形式**：双人对话体，配场景描述
- **生成策略**：围绕具体生活场景（点餐、问路、闲聊），用口语化短句，高频搭配优先
- **文章结构**：场景说明 → 对话正文 → 关键短语提炼
- **难度起点**：基础日常词汇，简单句式
- **适合人群**：想开口说英语的初学者

#### 职场英语（Workplace English）

- **内容形式**：邮件 / 会议纪要 / 职场对话，混合体裁
- **生成策略**：每篇聚焦一个职场场景（周会汇报、项目邮件、面试问答），强调正式表达与委婉措辞
- **文章结构**：场景背景 → 正文内容 → 实用句型 box
- **难度起点**：中等，假设用户有基础阅读能力
- **适合人群**：需要在工作中使用英语的职场人

#### 雅思备考（IELTS Prep）

- **内容形式**：学术短文，模拟雅思阅读段落风格
- **生成策略**：使用学术词汇（AWL 词表覆盖），长难句训练，段落逻辑严密
- **文章结构**：标准议论文结构（观点 → 论据 → 让步 → 总结）
- **难度起点**：中高，侧重 Band 6-7 词汇
- **适合人群**：正在备考雅思或需要学术英语的用户

#### 科技前沿（Tech & Science）

- **内容形式**：科技新闻 / 产品评测 / 概念解析风格的短文
- **生成策略**：围绕 AI、编程、硬件等话题，专业术语配即时解释
- **文章结构**：导语 → 正文 → 术语表（文末附关键术语释义）
- **难度起点**：中等，技术词汇多但句式不复杂
- **适合人群**：程序员、科技爱好者

#### 故事阅读（Story Time）

- **内容形式**：短篇故事 / 寓言 / 改编经典
- **生成策略**：叙事驱动，情节有趣味性和悬念，用故事包裹词汇学习
- **文章结构**：完整的叙事弧（开头 → 发展 → 高潮 → 结尾）
- **难度起点**：初级，但随故事推进词汇逐步丰富
- **适合人群**：喜欢阅读、希望在趣味中学习的用户

#### 自由主题（Free Topic）

- **内容形式**：根据用户描述自动匹配最合适的文体
- **生成策略**：用户输入关键词或一段描述，AI 判断最佳呈现形式
- **适合人群**：有明确兴趣方向的用户

### 自定义主题（Custom Topic）

用户可以创建完全自定义的主题，定义自己的学习方向。自定义主题与内置主题享有同等地位——同样参与难度梯度、嵌入式复习、设置覆盖等所有系统机制。

创建自定义主题时，用户需要配置：

| 字段 | 必填 | 说明 |
|------|------|------|
| 主题名称 | 是 | 如「医学英语」「法律合同」「游戏评测」 |
| 主题描述 | 是 | 一句话说明学习目标，同时作为生成 prompt 的上下文 |
| 内容形式 | 是 | 选择一种：对话体 / 文章体 / 故事体 / 邮件体 / 混合体 |
| 生成指令 | 否 | 用户自写的 prompt 片段，会拼接到系统 prompt 中。留空则由 AI 根据主题描述自动决定风格 |
| 难度起点 | 否 | 1-10，留空使用全局默认值 |
| 参考词汇 | 否 | 用户可粘贴一批希望优先出现的词汇，系统会在生成时优先编织 |

**自定义主题示例**：

```
主题名称：医学英语
主题描述：学习医疗场景下的英语表达，包括问诊对话、病历阅读、医学论文摘要
内容形式：混合体
生成指令：模拟真实医疗场景，每篇聚焦一个科室或病症，术语首次出现时括号注释
难度起点：5
参考词汇：diagnosis, symptom, prescription, chronic, acute, inflammation...
```

自定义主题在首页主题栏中排列在内置主题之后，支持编辑和删除。

### 主题配置参数

每个主题在系统层面有以下可配置项（用户可在设置页覆盖全局默认值）：

| 参数 | 说明 | 默认值 |
|------|------|--------|
| difficulty_start | 起始难度（1-10） | 随主题不同 |
| difficulty_step | 每篇文章难度递增幅度 | 0.3 |
| difficulty_max | 难度上限 | 8 |
| article_length | 目标字数 | 300 |
| review_ratio | 复习词占比（0-1） | 0.3 |
| review_window | 复习词选取范围（最近 N 篇文章的生词） | 5 |
| new_word_density | 新词密度（每百词中的新词数） | 3 |

## 内容生成引擎（Generation Engine）

### 生成原则

**趣味优先**

文章必须让人想读下去。无论什么主题，内容要有吸引力——对话要有性格，故事要有悬念，科技文要有洞察。枯燥的教材式文章是反模式。

**难度梯度**

用户在一个主题下生成的第 1 篇和第 10 篇，难度是不同的：
- 第 1-3 篇：舒适区，用户能读懂 85%+ 的内容，建立信心
- 第 4-7 篇：拉伸区，新词增多，句式变复杂，但不至于挫败
- 第 8 篇起：挑战区，接近真实语料难度

难度递增由系统自动计算，用户无需手动调整（但可在设置中覆盖）。

**嵌入式复习（Review Weaving）**

这是系统的核心机制。生成新文章时，系统会：

1. 从用户生词本中筛选出需要复习的词（优先选熟练度为「陌生」和「模糊」的）
2. 按照复习窗口（默认最近 5 篇文章的生词）选取候选词
3. 按复习词占比（默认 30%）计算本篇应编织的旧词数量
4. 将这些词作为约束条件传给大模型，要求自然地融入新文章

用户感知到的是"咦，这个词之前见过"——而不是"又在背这个词"。复习发生在新场景中，记忆通过语境的变化得到强化。

### 生成流程

```
用户选择主题
    ↓
系统计算当前难度等级（基于该主题下已生成的文章数）
    ↓
系统从生词本中筛选待复习词
    ↓
组装 Prompt（主题策略 + 难度参数 + 复习词列表 + 内容形式要求）
    ↓
调用 Claude API 流式生成
    ↓
前端流式渲染，生成完成后进入阅读页
```

## 数据模型

所有业务数据按 user_id 隔离，存储在 SQLite 文件数据库中。

```
User
  ├── id
  ├── email             // 唯一，用于登录
  ├── password_hash     // bcrypt 哈希
  ├── created_at
  └── last_login_at

Topic
  ├── id
  ├── user_id           // 所属用户（内置主题在用户注册时复制一份到该用户名下）

  ├── description
  ├── content_format    // dialogue | essay | story | email | mixed | free
  ├── is_builtin        // 是否为内置主题（内置主题不可删除，但参数可覆盖）
  ├── prompt_template   // 生成 prompt 模板（内置主题有系统默认值，自定义主题由用户填写）
  ├── reference_words   // 参考词汇列表（自定义主题可选）
```

### 大模型配置

系统支持用户自选大模型来驱动内容生成：

| 供应商 | 说明 |
|--------|------|
| Anthropic（默认） | 使用 Claude API，系统提供默认 key 或用户填入自己的 |
| OpenAI | 使用 OpenAI API，用户需填入自己的 API Key |
| 自定义 | 任何兼容 OpenAI Chat Completions 格式的 API 端点（如 Ollama、DeepSeek、Azure OpenAI） |

用户在设置页配置后，所有文章生成、划词翻译、例句翻译都走用户选择的模型。系统 prompt 保持不变，仅切换底层调用。

### 语音朗读（TTS）

划词即朗读：用户选中单词或句子后，系统自动朗读选中内容。

| 方案 | 说明 | 适用场景 |
|------|------|----------|
| Web Speech API | 浏览器内置，零成本，无需网络请求 | 默认方案，兼容性好 |
| Edge TTS | 微软 Edge 的在线 TTS，音质优于浏览器内置，免费 | 用户追求更好发音质量时切换 |

交互细节：
- 划选单词：自动朗读该单词
- 划选句子/短语：自动朗读整段选中内容
- 生词卡片上的 🔊 按钮：点击朗读单词
- 口音跟随设置页的偏好（美式 / 英式）

## 技术方向

- 前端 Next.js + React，后端 Next.js API Routes
- 数据库 SQLite（通过 better-sqlite3 或 Drizzle ORM 访问），单文件部署
- 用户认证：邮箱 + 密码注册登录，session 存 cookie，密码 bcrypt 哈希
- AI 内容生成：后端统一代理，根据用户配置的 llm_provider 分发到对应 API
- TTS：前端调用 Web Speech API（默认）或 Edge TTS
- 响应式设计，优先桌面端体验

## 设计基调

- 阅读器风格——像翻开一本排版精良的书
- 大量留白，行距宽松，字体清晰
- 交互轻量：划词即标记，点击即发音
- 配色温暖克制，生词高亮用柔和底色融入阅读
- 复习词用微妙的视觉区分（如细虚线下划线 + `⟳` 标记），让用户意识到"这个词见过"但不喧宾夺主

## 不做的事

- 不做独立的背单词 / 闪卡功能（复习通过文章自然完成）
- 不做社交功能（分享、排行榜、学习小组）
- 不做听力、口语练习
- 不做考试真题训练
- 不做移动端 App（先做好 Web）
